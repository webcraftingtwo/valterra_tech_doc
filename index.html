System Architecture

The application is intentionally simple — two standalone HTML files with no build process, no dependencies to install, and no server-side code. This architecture was chosen for reliability in a resource-constrained environment where IT support is limited.

📱
valterra-slam-v15.html — Miner App
Single HTML file. Vanilla JavaScript. No framework. Runs in any mobile browser. All logic, styles, and structure in one file for simplicity of deployment and maintenance.
Netlify · index.html
🖥️
dashboard.html — SHE Dashboard
Single HTML file. Vanilla JavaScript plus Chart.js for analytics. Role-gated — supervisor roles only. Real-time data via Supabase queries.
Netlify · dashboard.html
🗄️
Supabase — Backend
PostgreSQL database with Row Level Security. Supabase Auth for user management. No custom backend code — all data access via Supabase JS client with RLS policies.
Free Tier · Sufficient for pilot

Static file hosting. Both HTML files deployed by drag-and-drop. HTTPS included automatically. No server configuration required.

Authentication & Security

Auth method: Supabase email/password auth. Badge IDs are mapped to email format: badge.BADGEID@valterra-unki.com. These are system-generated addresses — no real email is sent or received.

User roles defined in auth metadata:

she_managerSHE ManagerFull procedureYes + Password Reset
adminAdministratorFull procedureYes + Password Reset

Row Level Security (RLS): All tables have RLS enabled. Authenticated users can insert their own records. All authenticated users can read all records (required for dashboard and supervisor functions). Updates are restricted to specific tables (sos_alerts, ground_alerts). The audit_log table is read-only via the app — writes happen only via the database trigger function running as SECURITY DEFINER.

API keys: Only the anon (public) key is used in the client-side code. The service role key is never exposed to the browser. The anon key with RLS provides appropriate data access control.

04
Key Safety Logic — How Critical Enforcement Works

2-Hour Lapse Enforcement: When the miner enters a blast time in Phase 1, the app calculates elapsed time in minutes using the current device clock. If elapsed time is less than 120 minutes, lapseCleared = false. The nextPhase() function checks this flag before allowing Phase 1 to advance. A live countdown refreshes every minute. The flag rechecks on each minute tick.

Gas Validation: Legal limits enforced — CO ≤ 50ppm, NO₂ ≤ 3ppm, O₂ ≥ 19.5%, H₂S ≤ 10ppm. On Phase 3 advancement, gasReadingsValid() is called. If any reading fails, the failure banner shows with retake and abort options. Gas readings cannot be marked safe by the miner — the comparison is done in code against hardcoded legal limits. Input bounds prevent impossible readings (CO max 500, NO₂ max 50, O₂ max 25, H₂S max 100).

Critical Step Locking: The CRITS object maps phase indices to arrays of critical step element IDs. On Next tap, nextPhase() checks every critical step ID for the checked CSS class. Any missing critical step blocks advancement and highlights the unchecked items. This cannot be bypassed from the UI.

FOG Light Block: If fogLightStatus === 'red', the ground alert modal opens immediately and gasReadingsValid() returns false. The procedure cannot advance. A ground alert record is saved to the ground_alerts table immediately — even before the miner confirms. The modal requires the miner to tap Confirm before the sent screen shows, but the DB write happens on the first sendGroundAlert() call.

SOS Reliability: The sendSOS() function wraps the database insert in try/catch. If the database insert fails for any reason, the sent confirmation screen still shows. The SOS UI response is never blocked by a database error.

Offline Mode: navigator.onLine is checked at submit time. If offline, the payload is serialised to localStorage under valterra_offline_queue. The online event listener triggers syncOfflineQueue() when connectivity is restored. Each queued item tracks a retry count — after 3 failed syncs the item moves to valterra_failed_queue and is not retried further.

05
Deployment & Configuration

Supabase credentials are hardcoded at the top of both HTML files:

Both files — top of <script> block
const SUPABASE_URL = 'https://yourproject.supabase.co';
const SUPABASE_KEY = 'your-anon-key-here';

Netlify deployment: Drag both files into the Netlify dashboard. Rename valterra-slam-v15.html to index.html. The dashboard file stays as dashboard.html. No build step, no configuration files needed.

Supabase Auth configuration: Email confirmations must be disabled in Supabase Auth settings (since badge email addresses are not real inboxes). Set Site URL to your Netlify URL. Disable email confirmation under Auth → Settings → Email.

Session management: Supabase JWT tokens expire after 1 hour by default. The app checks and silently refreshes the session every 3 minutes via db.auth.refreshSession(). If the refresh fails, the miner sees a re-authentication modal and their draft is preserved in localStorage.
